# Lab 3 三合一舆情决策助手设计

## 1. 目标

在现有 Lab 1 数据采集和 Lab 2 情感/观点分析基础上，实现一个本地运行的 Streamlit 决策支持助手。助手面向课程演示中的运动队传播人员或舆情分析人员，提供三项相互衔接的能力：自动简报、交互问答和方案建议。

系统只辅助信息整理与方案比较，不替代人工决策，不自动发布内容，也不把小样本案例分析包装成总体民意或因果预测。

## 2. 已确认的产品边界

- 使用独立的 Streamlit 本地网页，不改造 MediaCrawler 的 React/FastAPI 界面。
- 只读取当前 8 场 active 比赛及 Lab 2 固定数据格式，不提供文件上传。
- 在线模式复用 Lab 2 的 DeepSeek OpenAI 兼容接口；无密钥、网络异常或模型异常时自动使用离线兜底。
- 采用“证据优先”的模块化架构：程序负责数据校验、统计、筛选和引用，LLM 只负责解释、组织问答和生成候选方案。
- 正文与评论始终分开统计；可在同一页面对照，但不直接混合计算比例。
- 所有输出强制展示样本范围、证据记录和数据局限。

## 3. 用户与核心任务

主要用户是模拟的运动队传播或舆情管理人员。典型任务包括：

1. 选择单场比赛、胜球组、输球组或胜负对比，快速了解赛后 24 小时舆情。
2. 对当前范围继续追问，例如“输球后的评论是否都是负面的”。
3. 根据传播目标获取多个候选行动方案，比较证据、收益、风险和核验事项。
4. 在人工确认后导出一份可用于汇报或内部讨论的 Markdown 简报。

## 4. 总体架构

```text
Lab 1 events.json
Lab 2 sentiment_report.json + posts_sentiment.jsonl + comments_sentiment.jsonl
                              │
                              ▼
                   数据加载与契约校验
                              │
                              ▼
                范围筛选与确定性证据引擎
                              │
              ┌───────────────┼───────────────┐
              ▼               ▼               ▼
          自动简报服务     交互问答服务     方案建议服务
              └───────────────┼───────────────┘
                              ▼
                   DeepSeek / 离线兜底
                              │
                              ▼
             Streamlit 展示、证据展开与人工确认
```

各层职责如下：

- `data_loader`：读取事件、聚合报告和逐条 JSONL；建立统一记录索引；拒绝结构错误的数据。
- `evidence`：按照范围和正文/评论来源筛选数据，生成统计事实、代表观点、引用和覆盖警告。
- `briefing`：用同一证据包生成结构化简报；在线模式润色叙述，离线模式使用确定性模板。
- `qa`：把问题映射到当前证据范围；在线模式生成受约束回答，离线模式支持预设问题和保守兜底。
- `strategy`：根据受众与传播目标生成三种候选方案，并输出依据、收益、风险和人工核验项。
- `llm_client`：封装 DeepSeek 调用、结构校验、超时和自动降级，不让 UI 直接依赖 SDK。
- `app`：只负责 Streamlit 状态、输入控件和展示，不在页面代码中重复业务计算。

## 5. 数据契约与分析范围

数据源：

- `王楚钦舆情分析_Lab1/03_说明与配置/events.json`
- `王楚钦舆情分析_Lab2/01_输出/sentiment_report.json`
- `王楚钦舆情分析_Lab2/01_输出/posts_sentiment.jsonl`
- `王楚钦舆情分析_Lab2/01_输出/comments_sentiment.jsonl`
- `王楚钦舆情分析_Lab2/02_质量报告/lab3_ingestion_check.json`

应用启动时验证以下条件：

- 仅载入 `active=true` 的 8 场比赛；记录的 `event_id` 必须存在于事件配置。
- `match_result`、`content_type`、情感极性、领域情绪和议题字段必须符合 Lab 2 契约。
- `hours_after_event` 必须位于赛后 24 小时范围。
- 帖子和评论 ID 唯一；评论可保留父帖关联。
- Lab 3 摄入检查不得存在 error。已知覆盖 warning 继续展示而不是当作格式错误。

范围对象统一表示为：

- `single_event`：一场比赛；
- `win_group`：4 场胜利事件；
- `loss_group`：4 场失利事件；
- `win_loss_comparison`：胜负组对照。

组级指标优先读取 Lab 2 已计算的事件等权汇总；单场指标读取聚合报告并以逐条数据补充证据。不得在 Lab 3 中把全部记录简单混合后重新解释为组级比例。

## 6. 页面与交互

### 6.1 全局侧边栏

- 分析范围：单场、胜组、负组或胜负对比。
- 单场模式下选择具体赛事。
- 证据来源：正文与评论对照、仅正文或仅评论。
- 目标受众：球迷、媒体、运动队内部。
- 当前模式：DeepSeek 在线或离线兜底。
- 数据状态：记录数、摄入检查状态及覆盖警告。

范围、来源或受众变化后，三个页签共享新上下文；问答历史自动清空，避免引用旧范围。

### 6.2 自动简报

页面先展示确定性指标，再展示生成文本：

- 样本量、平均情感分和正/负/中比例；
- 主导情绪、主要议题与 24 小时时间线；
- 正文与评论差异；
- 支持、批评和理性分析等代表观点；
- “事实摘要—关键观察—决策关注—数据局限”四段式简报；
- 可展开的证据卡，包含记录 ID、来源、赛事、原文摘录、情感、议题和置信度；
- Markdown 下载。

### 6.3 交互问答

提供预设问题和自由提问。回答统一包含：

1. 数据事实；
2. 合理解释；
3. 证据引用；
4. 不能由数据支持的结论。

回答只能使用当前证据包。引用使用稳定的记录 ID，前端可展开对应原文。对于范围外、因果预测、私人信息或数据无法回答的问题，助手明确拒绝推断并说明原因。

离线模式覆盖一组确定性预设问题，包括胜负情感差异、正文/评论差异、主要议题、输球是否等于负面、代表观点和覆盖局限。自由问题无法可靠匹配时返回可用问题列表，不伪造答案。

### 6.4 方案建议

用户选择传播目标：回应争议、稳定球迷情绪、准备媒体简报或内部复盘。系统固定给出三类不同力度的候选方案，而不是输出单一“最佳答案”。每个方案包含：

- 行动名称、具体建议和适用时机；
- 所依据的事实与原文记录；
- 可能收益；
- 传播风险；
- 需要人工核实的信息；
- 风险等级和“非预测”提示。

方案不得包含自动发布按钮。页面底部提供人工选择、备注与最终理由输入区；人工内容只保存在 Streamlit 会话中，可随简报一起下载。

## 7. 证据与生成约束

证据引擎先生成结构化 `EvidencePacket`，至少包含：范围、来源、样本量、事件覆盖、确定性指标、Top 议题、代表记录和局限。所有 LLM 请求只接收该证据包和用户明确输入，不发送无关记录。

生成提示固定要求：

- 不修改或重新计算输入数字；
- 不声称数据代表微博总体舆情；
- 不把相关性描述为因果关系；
- 不使用“全网都在”“一定会”等绝对表达；
- 每个关键判断引用证据 ID；
- 方案只能是候选项，最终决定归人工；
- 输出满足定义好的 JSON 结构，解析失败则重试一次，之后降级为离线输出。

## 8. 在线与离线模式

在线模式优先从环境变量读取 `DEEPSEEK_API_KEY`，兼容 Lab 2 的 API 配置。调用设置有限超时，输出使用 JSON mode；模型名称和基础地址可通过环境变量覆盖，但默认值与 Lab 2 保持一致。

以下情况自动转入离线模式：

- 未配置 API key；
- 请求超时、网络失败或鉴权失败；
- 模型返回空内容；
- JSON 结构无法通过校验且一次重试仍失败。

降级只影响自然语言生成，不影响统计指标、证据卡和覆盖警告。页面清楚显示当前模式及降级原因，且允许用户再次尝试在线生成。

## 9. 错误处理

- 数据文件缺失或结构错误：应用停止核心功能，展示具体文件和错误，不用空数据继续生成。
- 已知稀疏事件：功能可用，但在指标、问答和方案中持续显示警告。
- 无匹配证据：返回“当前范围没有足够证据”，不调用 LLM 补全。
- API 故障：记录适合展示的简短原因并切换离线模式，不显示密钥、完整请求或敏感堆栈。
- Streamlit 会话切换范围：清除聊天和生成缓存，但保留数据加载缓存。

## 10. 隐私与安全

- 只使用已经脱敏的 Lab 1/2 数据，不恢复或展示个人身份。
- API key 仅从环境变量或本地、已忽略的配置读取，不写入仓库、日志或下载文件。
- 原文证据仅用于课程本地演示；界面继续标明数据来源和研究范围。
- 不生成针对具体普通用户的画像、处置建议或自动化互动行为。

## 12. 测试与验收标准

### 自动测试

- 数据加载：8 个 active 事件、45 条帖子、61 条评论，字段和 ID 校验正确。
- 证据范围：单场、胜组、负组和胜负对比不会混入错误事件；正文和评论分别计数。
- 覆盖警告：中国大满贯赢球事件零评论、新加坡输球事件仅 1 条评论时持续出现。
- 自动简报：离线输出包含事实、观察、关注点、局限和证据 ID，数字来自证据包。
- 问答：预设问题正确引用当前范围；范围外问题保守拒答。
- 方案建议：始终生成三个不同候选项，并包含依据、收益、风险、核验项和人工决策声明。
- LLM 客户端：成功、超时、无密钥、无效 JSON 和重试失败均得到确定性结果或安全降级。

### 集成与演示验收

1. 无 API key 启动 Streamlit，三项功能均可展示离线结果。
2. 有 API key 时可生成在线简报、自由问答和方案建议。
3. 切换赛事后旧问答不会保留。
4. 巴黎奥运输球案例可回答“输球是否等于负面”，并引用输球后仍支持的评论。
5. 稀疏评论事件会显示警告，不输出无依据的确定结论。
6. Markdown 下载包含范围、指标、证据、候选方案、人工备注和局限。

## 13. 非目标

- 不重新运行 Lab 1 采集或 Lab 2 情感标注。
- 不训练新模型，不实现向量数据库或通用 RAG 平台。
- 不接收任意上传数据，不支持多项目租户。
- 不做舆情变化的数值预测，不声称方案会导致确定结果。
- 不自动发布微博、发送消息或替代人工决策。
